# コンポーネントシステム

このドキュメントは、hidetaka.devのコンポーネントシステムとデザインシステムの概要をまとめたものです。

## 設計原則

### Pure Component
- すべてのUIコンポーネントはpure componentとして設計する
- propsのみに依存し、副作用を持たない
- データフェッチやAPI呼び出しを含まない
- 他のページやコンテキストで再利用可能

### 再利用性
- 汎用的な設計を心がける
- 特定のページやコンテキストに依存しない
- 必要に応じてvariantやpropsでカスタマイズ可能

### 型安全性
- TypeScriptで型定義を明確化
- propsの型を明示的に定義
- オプショナルなpropsには適切なデフォルト値を設定

### アクセシビリティ
- 適切なsemantic HTMLを使用
- aria-labelなどのアクセシビリティ属性を適切に設定
- キーボードナビゲーションを考慮

## ディレクトリ構造

```
src/components/
├── ui/              # 再利用可能なUIコンポーネント（デザインシステム）
├── tailwindui/      # Tailwind UIベースのコンポーネント
├── containers/      # ページ固有のコンテナコンポーネント
└── [feature]/      # 機能別のコンポーネント
```

## UIコンポーネント（`src/components/ui/`）

デザインシステムとして整理された再利用可能なpure componentの集合です。

### コンポーネントの種類

#### 基本UI要素
- バッジ、タグ、ボタンなどの基本UI要素
- 日付表示、画像表示などの表示コンポーネント

#### レイアウト要素
- セクションヘッダー、背景装飾などのレイアウト関連コンポーネント

#### カード要素
- 各種カードコンポーネント（スタックアイテム、プロジェクトなど）

### コンポーネント設計パターン

#### Props設計
- 必須propsとオプショナルpropsを明確に分離
- variantパターンでスタイルバリエーションを提供
- className propで追加のスタイリングを許可

#### スタイリング
- Tailwind CSSを使用
- ダークモード対応を標準で実装
- レスポンシブデザインを考慮

## コンポーネントの追加・変更

### 新しいコンポーネントの追加
1. `src/components/ui/`ディレクトリに新しいコンポーネントファイルを作成
2. pure componentとして実装（propsのみに依存）
3. TypeScriptで型定義を明確化
4. 必要に応じてvariantやsizeなどのpropsを提供

### 既存コンポーネントの変更
- 既存のAPIを壊さないように注意
- 後方互換性を保つ
- 大きな変更の場合は、新しいvariantやpropsとして追加を検討

## 使用ガイドライン

### コンポーネントの選択
- 既存のUIコンポーネントで要件を満たせる場合は、それを使用する
- 新しいコンポーネントが必要な場合は、pure componentとして実装し、`src/components/ui/`に追加

### カスタマイズ
- コンポーネントのvariantやpropsでカスタマイズできない場合は、新しいコンポーネントを作成する
- ページ固有のスタイリングは、コンテナコンポーネントで行う

### 命名規則
- コンポーネント名はPascalCase
- ファイル名はコンポーネント名と同じ
- 明確で説明的な名前を使用

## 注意事項

1. **コンポーネントの責務を明確に** - 各コンポーネントは単一の責務を持つ
2. **過度な抽象化を避ける** - 必要以上に汎用的にしない
3. **一貫性を保つ** - 既存のコンポーネントと一貫したAPI設計
4. **ドキュメント化** - 複雑なコンポーネントは使用例をコメントで記載
